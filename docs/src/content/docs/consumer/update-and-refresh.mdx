<Aside type="caution" title="Coming from npm? Read this first.">
`apm update` does **not** refresh your project's package dependencies. It
updates the `apm` CLI binary itself (think `npm update -g npm`).

To refresh the APM packages locked in your project, use
`apm install --update`. To preview what would change, use `apm outdated`.
</Aside>

There are two things that drift out of date in an APM project:

1. The `apm` CLI binary on your machine.
2. The package versions pinned in `apm.lock.yaml`.

They are managed by separate commands. Pick the right one.

## Update the apm CLI binary

```bash
apm update
```

Downloads the latest release of the `apm` CLI from the official installer
URL and replaces the binary in place. Use `--check` to see whether an
update exists without installing it:

```bash
apm update --check
```

That is the entire surface. It does not read `apm.yml`. It does not touch
`apm_modules/`. It does not modify the lockfile.

## Inspect what is outdated in your project

```bash
apm outdated
```

Compares each entry in `apm.lock.yaml` against its remote source and
reports which dependencies have new commits or new tags available. It
does not modify anything.

Useful flags:

- `-v, --verbose` -- show available tags for outdated tag-pinned deps.
- `-g, --global` -- check user-scope dependencies under `~/.apm/`
  instead of the current project.
- `-j, --parallel-checks N` -- cap concurrent remote checks (default 4;
  `0` for sequential).

Run this before you refresh, so you know what you are about to bump.

## Refresh project dependencies

```bash
apm install --update
```

Re-resolves every dependency in `apm.yml` to its latest matching Git
reference, rewrites `apm.lock.yaml` with the new commit SHAs and content
hashes, and re-deploys the resolved primitives into your harnesses.

This is the command that *actually changes versions in your project*. It
is the `npm update` equivalent for APM packages.

You can scope a refresh by combining `--update` with the same flags you
already use for `apm install`, for example:

```bash
apm install --update --only apm           # refresh APM packages, leave MCP alone
apm install --update --target claude       # only redeploy to Claude
apm install --update --dry-run            # preview the bump without writing
```

`--dry-run` is the safer cousin of `apm outdated` when you want to see the
exact resolution APM would commit, including transitive changes.

## Plain `apm install` does not refresh

```bash
apm install
```

With no flags, `apm install` is a *sync*: it reads `apm.yml` plus the
existing `apm.lock.yaml` and reproduces exactly the versions the lockfile
already pins. New dependencies you have added to `apm.yml` get resolved
and locked; existing ones stay on their locked SHAs.

That is the behavior you want for CI and for new contributors cloning
the repo (see [Promise 1: Portable by manifest](../../concepts/the-three-promises/)).

There is no `--frozen` flag. To opt out of refreshing, just omit
`--update`.

## When to use which

| You want to... | Run |
|---|---|
| Update the `apm` CLI itself | `apm update` |
| See if a CLI update exists | `apm update --check` |
| See which deps have new versions | `apm outdated` |
| Preview a dep refresh | `apm install --update --dry-run` |
| Bump deps and rewrite the lockfile | `apm install --update` |
| Reproduce the locked versions exactly | `apm install` |
